I chose this issue because it sits at the center of LLM gateway infrastructure engineering: routing and failover logic between multiple providers serving the same underlying model family. LiteLLM sits in front of dozens of LLM providers and is used in production RAG and agent systems, so a routing bug here directly breaks multi-turn tool-calling conversations for real users the moment a fallback is triggered.

This builds directly on the exception-handling and error-semantics work from my first contribution (prebid/salesagent #1433), but applies it to a different layer of the stack: instead of mapping SDK exceptions to HTTP status codes, this issue is about a stateful protocol detail (an opaque, endpoint-bound signature) that must be sanitized when a request crosses a provider boundary. I want to strengthen my understanding of multi-provider LLM routing, since it is a core competency for backend/AI-infrastructure roles working with LLM gateways.

## Understanding the Issue

### Problem Description

Gemini 3 models return a "thought signature" (`thoughtSignature`) attached to tool-call responses, used by Google to preserve reasoning continuity across multi-turn tool-calling conversations. This signature is bound to the endpoint that minted it: a signature produced by a request to Vertex AI is not valid when replayed to Google AI Studio's Gemini API, and vice versa.

LiteLLM's Router supports automatic fallback between deployments in a model group (e.g. falling back from a Vertex AI Gemini deployment to a Google AI Studio Gemini deployment when the first one errors). When this fallback happens mid-conversation, the message history — including any thought signatures generated by the original endpoint — is forwarded unchanged to the new endpoint via `run_async_fallback()` in `litellm/router_utils/fallback_event_handlers.py`. The new endpoint rejects the foreign signature as corrupted, and the fallback itself fails with an HTTP 400 error, defeating the purpose of the fallback.

### Expected Behavior

When the Router falls back across the `vertex_ai` <-> `gemini` (Google AI Studio) boundary, any thought signature from the original provider should be treated as if it were **missing** for the new target endpoint and replaced with Google's documented placeholder value (`skip_thought_signature_validator`, base64-encoded), rather than being replayed as-is.

### Current Behavior

The codebase already has this exact substitution mechanism, but only for the case where a signature is **absent**. `_get_thought_signature_from_tool()` and `convert_to_gemini_tool_call_invoke()` in `litellm/litellm_core_utils/prompt_templates/factory.py` insert a dummy signature (`_get_dummy_thought_signature()`, `dummy_data = b"skip_thought_signature_validator"`) only when no signature is found on the tool call and the target model is Gemini 3+. There is no code path that detects a signature is present but **foreign** (minted by a different provider) and needs the same substitution. The fallback handler in `fallback_event_handlers.py` has no awareness of `custom_llm_provider` for either the original or the fallback deployment, so it cannot even detect that a provider-boundary crossing occurred.

### Affected Components

- `litellm/router_utils/fallback_event_handlers.py` — `run_async_fallback()` (lines 85-165): performs the fallback but does not compare source/target `custom_llm_provider`
- `litellm/litellm_core_utils/prompt_templates/factory.py` — `_get_thought_signature_from_tool()` (lines 1207-1250), `_get_dummy_thought_signature()` (lines 1253-1263), `convert_to_gemini_tool_call_invoke()` (lines 1266-1374): existing dummy-signature substitution, currently missing-signature-only
- `litellm/llms/vertex_ai/gemini/vertex_and_google_ai_studio_gemini.py` — `VertexGeminiConfig._is_gemini_3_or_newer()`, `_forward_gemini_function_call_id()`: existing provider/model-version detection helpers that a fix would need to reuse
- `litellm/types/router.py` — `LiteLLMParamsTypedDict` (line 359): where each deployment's `custom_llm_provider` is stored and could be read during fallback
- `tests/test_litellm/llms/vertex_ai/gemini/test_thought_signature_in_tool_call_id.py` and `tests/test_litellm/llms/vertex_ai/gemini/test_vertex_ai_gemini_transformation.py`: existing test coverage for the missing-signature dummy substitution pattern that a fix's tests would extend
