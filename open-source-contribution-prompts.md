# Open Source Contribution Prompts
## Phases I – IV | Use in Serial Order

> **Global rule for all phases:** You are an AI assistant helping me write and commit code.
> Never add yourself as a co-author, contributor, or collaborator in any git commit message,
> PR description, or any file. Every commit must show only my name as the sole author.
> Do not include any "Co-Authored-By", "Generated by", or similar lines anywhere.

---

## Phase I — Explore & Select

```
I am doing an open source contribution as part of a course assignment.

Project: [UPSTREAM_REPO_URL]  e.g. https://github.com/org/repo
My fork: [FORK_URL]           e.g. https://github.com/YOUR_USERNAME/repo
Issue I selected: [ISSUE_URL] e.g. https://github.com/org/repo/issues/1433

Help me complete Phase I of my contribution:

1. Read the issue description carefully and explain the bug or feature request
   in simple terms — what is broken, what should happen instead, and what files
   are likely involved.

2. Explore the codebase to find:
   - The specific file(s) and line numbers mentioned in or relevant to the issue
   - Any existing patterns in the codebase that are similar to the fix needed
   - The CONTRIBUTING.md and any setup documentation

3. Update my Contribution README ([CONTRIBUTION_README_PATH]) with:
   - A "Why I Chose This Issue" section explaining my motivation
   - An "Understanding the Issue" section covering: problem description,
     expected behavior, current behavior, and affected components

Keep all content factual and based on what you find in the code.

IMPORTANT: Do not add yourself as a co-author or contributor anywhere.
All commits must show only my name as the sole author — no Co-Authored-By,
Generated-by, or similar lines in any commit message or file.
```

## Phase II — Reproduce & Plan

```
I am doing an open source contribution. Phase I is complete.

Project fork (local path): [LOCAL_REPO_PATH]  e.g. /Users/name/projects/repo
Issue: [ISSUE_URL]
Contribution README: [CONTRIBUTION_README_PATH]

Help me complete Phase II:

1. Set up the local development environment:
   - Read README.md and CONTRIBUTING.md to find setup steps
   - Install dependencies and verify the project runs
   - Document any setup challenges encountered

2. Create a working branch named fix-issue-[NUMBER] and push it to my fork.

3. Reproduce the bug without fixing it:
   - Find the exact code location causing the issue
   - Write or run a script/test that demonstrates the wrong behavior
   - Show clearly: what the current output is vs what it should be
   - The branch should remain on the buggy code at this point

4. Update the Contribution README with:
   - Environment Setup notes (steps taken, any errors and how resolved)
   - Steps to Reproduce (numbered steps anyone could follow)
   - Reproduction Evidence (branch link, bug location, findings)
   - Solution Approach using the UMPIRE framework:
     * Understand: restate the problem in your own words
     * Match: what existing pattern in the codebase solves this?
     * Plan: specific files and lines to change, step by step
     * Implement: branch link placeholder
     * Review: how you will check against contribution guidelines
     * Evaluate: what tests will confirm the fix works

Do NOT implement any fix yet. Phase II ends with a plan, not code.

IMPORTANT: Do not add yourself as a co-author or contributor anywhere.
All commits must show only my name as the sole author — no Co-Authored-By,
Generated-by, or similar lines in any commit message or file.
```

## Phase III — Build

```
I am doing an open source contribution. Phase II is complete.

Project fork (local path): [LOCAL_REPO_PATH]
Issue: [ISSUE_URL]
Working branch: fix-issue-[NUMBER]
Contribution README: [CONTRIBUTION_README_PATH]

My Phase II plan is already written in the Contribution README.
Help me complete Phase III by executing that plan:

1. Review CONTRIBUTING.md for:
   - Code style and linting rules
   - Test requirements (unit, integration, etc.)
   - Commit message format

2. Implement the fix from the plan:
   - Make the minimal changes needed — no refactoring beyond what the issue requires
   - Match the existing patterns in the codebase exactly
   - Do not add features or cleanup unrelated to the issue

3. Write tests:
   - Find the existing test file for the code you modified
   - Update any tests whose expectations were based on the buggy behavior
   - Add new tests that cover each case fixed by the change
   - Run the tests and confirm they all pass

4. Run the project's quality checks (linting, type checking, test suite)
   and fix any failures before committing.

5. Commit with a conventional commit message and push to the working branch.

6. Update the Contribution README with:
   - Implementation Notes: what was built, files modified, key commit links,
     challenges faced, approach decisions
   - Testing Strategy: what tests were added/updated and their results
   - Update the Implement line in the UMPIRE plan with the actual commit link
   - Update the Evaluate line with actual test results
   - Change Status to "Phase III Complete"

Only include changes relevant to the issue in the diff.

IMPORTANT: Do not add yourself as a co-author or contributor anywhere.
All commits must show only my name as the sole author — no Co-Authored-By,
Generated-by, or similar lines in any commit message or file.
```

## Phase IV — Submit & Iterate

```
I am doing an open source contribution. Phase III is complete.

Upstream repo: [UPSTREAM_REPO_URL]
My fork: [FORK_URL]
Working branch: fix-issue-[NUMBER]
Issue: [ISSUE_URL]
Contribution README: [CONTRIBUTION_README_PATH]

Help me complete Phase IV:

1. Run final pre-submission checks:
   - Show the full diff (only files changed should be relevant to the issue)
   - Run the test suite one final time and confirm it passes
   - Confirm no debug code, stray comments, or unrelated changes exist

2. Check for a PR template:
   - Look in .github/PULL_REQUEST_TEMPLATE.md
   - Look in CODEOWNERS to find who to tag for review

3. Prepare the PR content I will paste into GitHub:
   - PR title following the project's commit convention
   - PR description covering: what this PR does, why it was needed,
     the issue number (Closes #NUMBER), and an acceptance criteria checklist
   - A review request comment to paste after the PR is created,
     tagging the appropriate maintainer from CODEOWNERS

4. After I confirm the PR URL ([I WILL PROVIDE THIS AFTER OPENING THE PR]):
   Update the Contribution README with:
   - Pull Request section: PR link, one-paragraph description of contribution,
     maintainer feedback placeholder, Status: Awaiting review
   - Learnings & Reflections: technical skills gained, challenges overcome,
     what I would do differently next time
   - Change Status to "Phase IV Complete"
   - Commit and push the README to my contribution repo

The PR must go from my fork's branch to the upstream repo's main branch.
Do not open the PR automatically — prepare the content and I will paste it
into GitHub myself.

IMPORTANT: Do not add yourself as a co-author or contributor anywhere.
All commits must show only my name as the sole author — no Co-Authored-By,
Generated-by, or similar lines in any commit message or file.
```
